How it works

Three services, zero manual wiring

Features

Built for production

Not a toy scraper. Real concurrency control, crash recovery, and container isolation.

🔐

Atomic slot management

Lua scripts in Redis enforce per-domain and global concurrency limits atomically — no race conditions, no overcommit.

🐳

Full container isolation

Each job runs in its own Docker container with 1GB RAM cap, 2GB shared memory, and SYS_ADMIN for Chromium sandbox.

🛡️

Cloudflare bypass

Playwright with virtual display, randomized viewports, human-like mouse movement, and Turnstile click automation.

♻️

Crash recovery

On restart, orphan containers are killed before stale jobs re-queue. No double-spawn, no resource leak.

🔍

Zero-config discovery

Drop a Dockerfile into workers/mynewdomain/ and restart. The orchestrator picks it up automatically.

📊

Real-time dashboard

Flask UI with paginated job lists, result viewer, per-domain stats, and one-click clear by state.

Getting started

Up in three commands

1 — Clone & configure
# Clone the repository
git clone https://github.com/xuandich/Redis-Server.git
cd Redis-Server

# Copy and edit your environment
cp .env.example .env
2 — Start everything
# Start Redis + Orchestrator + Dashboard
# Worker images are built automatically if not cached
./start.sh

# Dashboard is live at:
→  http://localhost:5000
3 — Submit a crawl job
curl -X POST http://localhost:5000/api/submit-job \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://www.manomano.fr/p/example-product-123",
    "domain": "manomano"
  }'

# Response:
{
  "ret_key": "ret_manomano_f3a2b1c0-...",
  "status": "queued"
}
4 — Fetch the result
curl http://localhost:5000/api/job/ret_manomano_f3a2b1c0-...

{
  "status": "success",
  "http_code": 200,
  "html": "<!DOCTYPE html>...",
  "elapsed_ms": 4821,
  "domain": "manomano"
}

REST API

Full HTTP interface

All dashboard functionality is exposed as a JSON API. Build your own client on top.

  • POST /api/submit-job Submit a new crawl job. Returns ret_key for polling.
  • GET /api/job/<ret_key> Fetch result for a specific job by its return key.
  • GET /api/jobs/<state> Paginated list of jobs by state: queued · running · finished · failed
  • GET /api/stats Aggregate counts by state and domain, success rate.
  • GET /api/workers List of active RQ workers and their current job.
  • POST /api/clear_state/<state> Bulk-delete all jobs in a given state from Redis.
  • POST /api/cancel/<ret_key> Cancel a queued job before it starts.
  • POST /api/delete/<ret_key> Delete job state and result from Redis.

Configuration

Environment variables

Copy .env.example to .env and adjust. Per-domain overrides follow the _{DOMAIN} suffix pattern.

REDIS_HOST
default: redis
Redis hostname inside the Docker network.
JOB_TIMEOUT_DEFAULT
default: 120
Seconds before container.wait() kills the container. Override per-domain with JOB_TIMEOUT_MANOMANO=300.
RESULT_TTL
default: 3600
Seconds to keep job results in Redis before auto-expiry.
MAX_CONCURRENT_TOTAL
default: 10
Global hard cap on simultaneous running containers across all domains.
MAX_CONCURRENT_{DOMAIN}
default: 5
Per-domain container limit. E.g. MAX_CONCURRENT_MANOMANO=3.
PROXY_HOST_DIR
default: —
Host path to the Proxy folder. Mounted read-only into each worker container at /app/Proxy.
CONTAINER_MEM_LIMIT
default: 1g
Docker memory limit per worker container.
CONTAINER_SHM_SIZE
default: 2g
Shared memory for Chromium. Must exceed 512MB for stable browser operation.

Extensibility

Adding a new domain worker

The system is designed so that one developer can add a new scraper in under an hour.

Directory structure — workers/mynewdomain/
workers/
└── mynewdomain/
    ├── Dockerfile          # triggers auto-discovery
    ├── run.py              # entry point — reads URL, RET_KEY from env
    └── sourceCode/
        ├── main.py         # async process_single_request(request)
        ├── extractor.py    # your scraping logic
        └── requirements.txt

The worker reads URL, RET_KEY, REDIS_HOST from environment variables, and writes its result to result:{ret_key} in Redis before exiting. The orchestrator handles everything else.
